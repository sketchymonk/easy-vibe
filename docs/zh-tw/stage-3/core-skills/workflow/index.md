在前面的章節中，我們學習了如何使用 AI IDE 寫程式碼、如何用 Git 管理程式碼版本，以及如何設計和實現 API 介面。但當你面對一個真實的開發任務時，可能會遇到這些問題：

- 「這個專案有上千個檔案，我該從哪裡開始？」
- 「老闆讓我加一個新功能，但我不熟悉這部分的程式碼庫。」
- 「我不知道這個 bug 在哪裡，程式碼太多了。」
- 「我需要重構這堆程式碼，但我怕改壞東西。」

這些問題的本質是：**如何在真實的開發場景中高效地使用 AI 工具來完成工作？**

# 1. 了解 AI 的能力邊界

在我們開始使用 AI 輔助開發之前，首先需要了解 AI 能做什麼、不能做什麼。只有這樣才能建立正確的協作模式。

## 1.1 AI 擅長什麼

可以把 AI 想像成一個非常聰明但仍需要明確指令的助手。它可以根據你的描述快速生成程式碼骨架，也能在幾秒鐘內閱讀上千行程式碼，找到你需要的那部分。如果有明顯的語法錯誤或常見的安全漏洞，它也能幫你發現。重複性的工作，比如批次重新命名變數、格式化程式碼、生成文件註釋等，特別適合交給 AI 來做。

簡單來說，AI 擅長有明確規則、可以自動化的工作。

## 1.2 AI 不擅長什麼

但 AI 也有它的局限性。它不理解你的業務邏輯，除非你詳細告訴它，否則它不會知道你們公司的訂單流程是怎麼運作的。它也無法做出需要權衡取捨的技術選型或架構設計決策，因為這些取決於你的經驗和對專案的理解。AI 也不知道你們團隊的特殊約定，比如「所有 API 都必須有日誌記錄」或「錯誤碼必須使用列舉型別」。你需要設定這些規則或明確告訴它。

最重要的是，AI 生成的程式碼不能直接使用。你必須審查和測試它。它可能生成看起來正確但實際有問題的程式碼，也可能忽略某些邊界情況。

## 1.3 如何與 AI 協作

了解了 AI 的能力邊界之後，協作模式就清晰了：你負責決定做什麼、做出決策、確保品質；AI 負責執行具體的程式碼編寫工作、尋找資訊、發現明顯的問題。

這就像和一個初級開發者合作一樣。你告訴他們需要做什麼，他們實作，然後你審查程式碼。不同的是，AI 執行速度快得多，但判斷力比人類弱。

# 2. 不同專案類型的開發策略

不同類型的專案需要不同的開發方式和 AI 使用策略。選擇正確的策略可以大幅提高開發效率。

## 2.1 全新專案（從零開始）

**專案特點：**
- 沒有歷史包袱，可以自由設計
- 需要建立專案結構和程式碼規範
- 適合快速迭代和試錯

**推薦工作流程：**

**第一步：規劃專案結構**

在開始寫程式碼之前，先讓 AI 幫你規劃專案結構和技術選型：

```text
我想建立一個任務管理應用，功能如下：
- 使用者註冊和登入
- 建立、編輯和刪除任務
- 任務分類和標籤
- 任務提醒

請幫我：
1. 推薦適合的技術堆疊
2. 設計專案目錄結構
3. 規劃資料庫 schema
```

**第二步：建立基本框架**

根據規劃，讓 AI 建立基本的專案結構：

```text
根據我們剛才討論的規劃，幫我：
1. 建立專案目錄結構
2. 初始化設定檔（package.json、.env 等）
3. 建立基本的伺服器程式碼
```

**第三步：逐個實現功能**

按優先順序逐個實現功能模組：

```text
現在實現使用者註冊功能，需求如下：
- 使用電子郵件和密碼註冊
- 密碼加密儲存
- 電子郵件驗證
```

**要點：**
- 盡早建立程式碼規範，讓 AI 生成的程式碼遵循這些規範
- 每完成一個功能模組就測試和驗證
- 及時更新專案文件

## 2.2 成熟專案（大型既有程式碼庫）

**專案特點：**
- 程式碼庫龐大，有歷史約定
- 需要保持程式碼風格一致
- 修改必須考慮影響範圍

**推薦工作流程：**

**第一步：了解專案結構**

在修改程式碼之前，先讓 AI 幫你了解專案：

```text
這是一個電商專案，我需要新增優惠券功能。
請幫我：
1. 分析整體專案結構
2. 找到訂單相關的程式碼
3. 看看其他類似功能是如何實現的
```

**第二步：找到參考程式碼**

讓 AI 在專案中找到類似的實現作為參考：

```text
找出專案中其他促銷功能（如滿減、折扣）是如何實現的
```

**第三步：遵循既有風格**

讓 AI 按照既有程式碼的風格實現新功能：

```text
請參考滿減促銷的實現方式來實現優惠券功能。
保持相同的程式碼風格和目錄結構。
```

**要點：**
- 先了解再修改，不要破壞既有架構
- 保持程式碼風格一致
- 修改後測試相關功能

## 2.3 快速原型（驗證想法）

**專案特點：**
- 速度最重要，程式碼品質次要
- 用於驗證產品想法或技術方案
- 可能之後會丟棄或重寫

**推薦工作流程：**

**直接描述需求並快速實現：**

```text
建立一個簡單的待辦事項應用，需求如下：
- 新增、刪除和標記任務為已完成
- 本地儲存資料
- 介面簡單，能用就行
```

**快速迭代：**

```text
新增搜尋功能
切換為深色主題
新增任務分類
```

**要點：**
- 不用太擔心程式碼品質或規範
- 快速驗證想法，及時調整方向
- 如果原型成功，之後需要重構

## 2.4 維護專案（主要是修 bug）

**專案特點：**
- 程式碼已穩定，主要任務是修復問題
- 需要快速定位問題
- 修改必須小心，避免引入新問題

**推薦工作流程：**

**第一步：定位問題**

```text
使用者回饋：點擊「提交訂單」按鈕後，頁面卡住
主控台錯誤：TypeError: Cannot read property 'id' of undefined

請幫我：
1. 分析可能的原因
2. 找到相關的程式碼
```

**第二步：分析根本原因**

```text
檢查在什麼情況下會出現這個錯誤
檢查資料流向
```

**第三步：應用修復**

```text
修復這個問題，並且：
1. 新增防禦性程式碼避免類似問題
2. 新增錯誤提示改善使用者體驗
```

**要點：**
- 修復後全面測試，確保不影響其他功能
- 新增防禦性程式碼提高系統穩健性
- 記錄問題和解決方案以供未來參考

# 3. 常見開發任務的工作流程

在日常開發中，我們會遇到許多不同類型的任務。以下是幾個最常見的 AI 輔助工作流程。

## 3.1 開發新功能

**場景：** 產品經理給你一個新需求，你需要實現一個新功能。

**完整工作流程：**

**第一步：理解需求**（由你完成）

在開始寫程式碼之前，先釐清：
- 需要實現什麼功能？
- 輸入和輸出是什麼？
- 邊界條件和異常情況有哪些？
- 性能和安全要求是什麼？

**第二步：了解既有程式碼**（讓 AI 幫忙）

```text
我需要實現使用者評論功能。請幫我：
1. 檢查專案中是否有類似的東西
2. 找到使用者資料和文章資料是如何儲存的
3. 了解這個專案中的資料庫操作慣例
```

**第三步：制定實作計畫**（與 AI 一起）

```text
根據我們剛才的分析，實現評論功能需要：
1. 建立評論資料模型
2. 實現評論的 CRUD API
3. 在前端新增評論顯示和發佈功能

請幫我確認這個計畫是否合理，是否有遺漏
```

**第四步：實現功能**（AI 生成，你審查）

```text
根據計畫，先實現評論資料模型。
需求：
- 包含評論內容、作者、文章 ID 和建立時間
- 支援巢狀評論回覆
- 新增必要的索引
```

```text
現在實現評論 API 端點。
需求：
- 發佈評論
- 刪除評論（只能刪除自己的評論）
- 取得文章的所有評論
- 支援分頁
```

```text
最後實現前端評論元件。
需求：
- 顯示評論列表
- 評論發佈表單
- 支援回覆評論
```

**第五步：測試和驗證**（由你完成）

- 手動測試主流程
- 測試邊界情況，如空內容和過長內容
- 測試許可權控制，如無法刪除他人的評論
- 請 AI 幫你撰寫自動化測試

**真實對話範例：**

```text
你：我想新增一個使用者評論功能，讓使用者可以在文章上發佈評論

AI：好的，讓我先了解一下專案結構。
    [AI 分析專案程式碼]
    我看到專案已經有了使用者系統（models/user.js）
    和文章系統（models/article.js）。

    實現評論功能需要：
    1. 建立評論資料模型（models/comment.js）
    2. 新增評論相關的 API（routes/comment.js）
    3. 在文章詳情頁新增評論顯示和發佈功能

    我可以開始實現嗎？

你：好的，開始吧。先建立資料模型

AI：[生成 models/comment.js]
    我已經建立了評論資料模型，包含這些欄位：
    - id: 評論 ID
    - content: 評論內容
    - userId: 作者 ID
    - articleId: 文章 ID
    - parentId: 父評論 ID（用於巢狀回覆）
    - createdAt: 建立時間

    接下來要實現 API 嗎？

你：好的，實現 API
```

## 3.2 修復 Bug

**場景：** 使用者回報了一個問題，你需要快速定位並修復它。

**完整工作流程：**

**第一步：收集資訊**（由你完成）

記錄以下資訊：
- 錯誤訊息是什麼？
- 這個問題如何重現？
- 什麼時候開始出現的？
- 影響範圍有多大？

**第二步：定位問題**（讓 AI 幫忙）

```text
使用者回饋：點擊「提交訂單」按鈕後，頁面卡住
瀏覽器主控台錯誤：TypeError: Cannot read property 'id' of undefined
錯誤出現在 checkout.js 的第 45 行

請幫我：
1. 分析可能的原因
2. 找到相關的程式碼
3. 檢查資料流向
```

**第三步：分析根本原因**（與 AI 一起）

```text
檢查這個 user 物件從哪裡來的
檢查在什麼情況下 user 可能是 undefined
```

**第四步：實施修復**（AI 生成，你審查）

```text
修復這個問題，需求如下：
1. 新增防禦性程式碼檢查 user 是否存在
2. 如果 user 不存在，重新導向到登入頁面
3. 新增友善的錯誤提示
```

**第五步：驗證修復**（由你完成）

- 用原來的步驟重現問題，確認問題已解決
- 測試相關功能，確保沒有引入新問題
- 想想其他地方是否存在類似問題

**真實對話範例：**

```text
你：有使用者回報登入後重新整理頁面會被登出

AI：這通常是 token 儲存或驗證的問題。讓我檢查一下：
    1. token 儲存在哪裡，localStorage 還是 cookies？
    2. 頁面重新整理時 token 是否會重新載入？

    請告訴我哪個檔案包含 token 相關的程式碼

你：utils/auth.js

AI：[分析 utils/auth.js]
    我找到問題了：token 儲存在 sessionStorage 中，
    而 sessionStorage 在頁面重新整理時會被清除。

    解決方案：改用 localStorage

    你要我修改嗎？

你：好的，修改吧

AI：[修改程式碼]
    修改完成。token 現在儲存在 localStorage 中，
    所以頁面重新整理後不會丟失。

    我建議你測試以下項目：
    1. 登入後重新整理頁面，確認使用者保持登入狀態
    2. 關閉並重新開啟瀏覽器，確認使用者保持登入狀態
```

## 3.3 重構程式碼

**場景：** 程式碼變得難以維護，需要重構以提高程式碼品質。

**完整工作流程：**

**第一步：定義重構目標**（由你完成）

釐清以下問題：
- 你想解決什麼問題？（重複程式碼？結構混亂？性能問題？）
- 重構的範圍有多大？
- 可接受的風險等級是什麼？

**第二步：制定重構計畫**（與 AI 一起）

```text
這三個函式（getUserList、getArticleList、getCommentList）
包含大量重複的分頁、排序和篩選程式碼。

請幫我：
1. 分析重複的部分
2. 提出重構方案
3. 評估重構的風險
```

**第三步：準備安全網**（讓 AI 幫忙）

```text
在重構之前，請先幫我寫測試案例
確保重構後行為保持一致
```

**第四步：逐步重構**（AI 執行，你驗證）

```text
根據我們剛才討論的計畫，先提取共用的函式
```

```text
現在最佳化資料結構
```

```text
最後更新呼叫處
```

**第五步：驗證結果**（由你完成）

- 執行所有測試，確保通過
- 手動測試關鍵功能
- 檢查程式碼品質是否有所提升

**真實對話範例：**

```text
你：這三個函式有很多重複程式碼。你能幫我重構嗎？
    [貼上三個函式的程式碼]

AI：我分析了一下，主要差異是：
    - getUserList: 取得使用者列表
    - getArticleList: 取得文章列表
    - getCommentList: 取得評論列表

    它們的共同邏輯是分頁、排序和篩選。

    重構方案：
    1. 提取一個通用的 getList 函式
    2. 將不同的部分作為參數傳入

    這樣可以減少 80% 的重複程式碼。

    我可以開始實現嗎？

你：好的，但先幫我寫測試案例

AI：[生成測試]
    我已經為三個函式都寫了測試，
    涵蓋了主要的使用場景。

    我可以開始重構了嗎？

你：好的，開始重構

AI：[重構程式碼]
    重構完成。我建立了一個通用的 getList 函式，
    原來的三個函式現在都呼叫這個共用函式。

    請執行測試確認一切正常：npm test
```

## 3.4 程式碼審查

**場景：** 團隊成員提交了程式碼，你需要進行審查。

**完整工作流程：**

**第一步：快速了解變更**（讓 AI 幫忙）

```text
總結這個 PR 改了什麼
主要修改了哪些檔案
```

**第二步：檢查常見問題**（讓 AI 幫忙）

```text
檢查這個變更是否有：
1. 安全問題（SQL 注入、XSS 等）
2. 性能問題（N+1 查詢、記憶體洩漏等）
3. 明顯的 bug
4. 程式碼風格是否符合專案規範
```

**第三步：深入審查**（由你完成）

- 業務邏輯是否正確
- 是否考慮了邊界情況
- 是否有更好的實現方式
- 是否需要新增測試

**第四步：給出回饋**（由你完成）

- 指出需要修改的地方
- 解釋為什麼需要修改
- 提供改善建議

**真實對話範例：**

```text
你：幫我審查這個 PR
    [提供 PR 連結或程式碼]

AI：我審查了這個變更，發現了以下問題：

    1. 安全問題：
       - 第 23 行：使用者輸入未轉義，可能有 XSS 風險
       - 第 45 行：SQL 查詢使用字串拼接，有注入風險

    2. 性能問題：
       - 第 67 行：在迴圈中查詢資料庫，造成 N+1 問題

    3. 程式碼品質：
       - handleSubmit 函式太長（200 行），建議拆分
       - 缺少錯誤處理

    你需要我詳細解釋其中任何一個嗎？

你：詳細解釋一下 SQL 注入問題

AI：在第 45 行：
    ```javascript
    const query = `SELECT * FROM users WHERE name = '${userName}'`;
    ```

    這裡直接將使用者輸入拼接到 SQL 語句中。
    如果使用者輸入 `' OR '1'='1`，就會變成：
    ```sql
    SELECT * FROM users WHERE name = '' OR '1'='1'
    ```
    這會返回所有使用者資料。

    正確的做法是使用參數化查詢：
    ```javascript
    const query = 'SELECT * FROM users WHERE name = ?';
    db.query(query, [userName]);
    ```

    你需要我修復嗎？
```

# 4. 建立專案知識庫

為了幫助 AI 更好地理解你的專案，建議在專案中建立一個知識庫。這樣 AI 就能按照你的慣例和習慣來工作。

## 4.1 建立專案描述檔案

在專案根目錄建立一個 `CLAUDE.md` 或 `AGENTS.md` 檔案，記錄關鍵的專案資訊：

```markdown
# 專案概覽

## 專案摘要
這是一個線上學習平台，提供課程管理、使用者學習、作業提交等功能。

## 技術堆疊
- 前端：React 18 + TypeScript + Vite
- 後端：Node.js + Express + PostgreSQL
- 部署：Vercel（前端）+ Railway（後端）

## 專案結構
```
src/
├── components/     # React 元件
├── pages/          # 頁面元件
├── api/            # API 呼叫
├── utils/          # 工具函式
└── types/          # TypeScript 型別定義
```

## 程式碼規範
- 使用 ESLint 和 Prettier 格式化程式碼
- 元件檔案使用 PascalCase（如 UserProfile.tsx）
- 工具函式使用 camelCase（如 formatDate.ts）
- 常數使用 UPPER_SNAKE_CASE（如 API_BASE_URL）

## 開發流程
1. 從 main 建立功能分支
2. 開發完成後提交 PR
3. 程式碼審查通過後合併

## 常用指令
- 啟動開發伺服器：`npm run dev`
- 執行測試：`npm test`
- 生產建置：`npm run build`
- 格式化程式碼：`npm run format`

## 注意事項
- 所有 API 呼叫都必須包含錯誤處理
- 使用者輸入必須驗證和轉義
- 資料庫操作使用參數化查詢，避免 SQL 注入
- 敏感資訊（密碼、token）不得寫入日誌

## 資料庫 Schema
- users: 使用者表（id, email, password_hash, created_at）
- courses: 課程表（id, title, description, teacher_id）
- enrollments: 選課表（id, user_id, course_id, enrolled_at）
```

## 4.2 記錄常見問題和解決方案

在專案中建立 `docs/troubleshooting.md` 記錄常見問題：

```markdown
# 常見問題

## 開發環境問題

### 問題：npm install 失敗
**原因：** Node 版本不相容
**解決方案：** 使用 Node.js 18 或更高版本

### 問題：資料庫連線失敗
**原因：** 環境變數未設定
**解決方案：** 將 .env.example 複製為 .env 並填入資料庫連線資訊

## 功能問題

### 問題：使用者登入後重新整理頁面會被登出
**原因：** token 儲存在 sessionStorage 中
**解決方案：** 改用 localStorage

### 問題：圖片上傳失敗
**原因：** 檔案大小超過限制
**解決方案：** 在前端新增檔案大小檢查，限制為 5MB
```

## 4.3 維護技術決策記錄

建立 `docs/decisions/` 目錄記錄重要的技術決策：

# ADR-001：選擇 PostgreSQL 作為資料庫

## 狀態
已接受

## 背景
專案需要選擇一個關聯式資料庫。候選方案有 MySQL 和 PostgreSQL。

## 決策
選擇 PostgreSQL

## 理由
1. 更好的 JSON 支援，適合儲存課程內容
2. 更強大的全文搜尋能力
3. 團隊對 PostgreSQL 更熟悉

- 部署需要 PostgreSQL 環境

# 5. 提升 AI 協作效率的技巧

掌握一些實用技巧可以讓你與 AI 的協作更加高效。

## 5.1 描述問題時要清晰具體

**不好的描述：**
```text
這個功能有問題
幫我最佳化一下
```

**好的描述：**
```text
使用者點擊「提交」按鈕後，表單沒有提交
瀏覽器主控台報錯：Uncaught TypeError: Cannot read property 'value' of null
錯誤出現在 form.js 的第 23 行

這個列表載入很慢，有 1000 個項目
請幫我加分頁功能，每頁 20 個項目
```

**要點：**
- 提供具體的錯誤資訊
- 說明預期結果
- 給出相關的上下文

## 5.2 每次只做一件事

**不好的做法：**
```text
幫我實現登入、註冊、密碼找回、個人中心、
修改密碼和郵箱驗證
```

**好的做法：**
```text
先實現登入功能，需求如下：
- 電子郵件和密碼登入
- 記住登入狀態
- 錯誤提示

（完成後）現在實現註冊功能

（完成後）現在實現密碼找回功能
```

**要點：**
- 將大任務拆分為小任務
- 每完成一個任務就測試和驗證
- 確認沒有問題後再進行下一個

## 5.3 及時驗證結果

**不好的做法：**
- 讓 AI 連續修改 10 個檔案
- 最後才發現第一個修改就已經錯了
- 浪費大量時間

**好的做法：**
- 修改一個檔案後立即測試
- 確認沒有問題再繼續
- 發現問題就及時修正

**要點：**
- 小步前進，快速回饋
- 不要盲目信任 AI
- 保持對程式碼的掌控

## 5.4 善用上下文

**技巧一：引用之前的對話**
```text
根據我們剛才討論的計畫來實現
參考之前的 getUserList 函式
```

**技巧二：提供相關程式碼**
```text
這是既有的使用者模型程式碼：
[貼上程式碼]

請用相同的風格實現文章模型
```

**技巧三：說明專案背景**
```text
這是一個電商專案，使用 React + Node.js
已經有了使用者系統和商品系統
現在我們需要新增購物車功能
```

## 5.5 儲存有用的對話

**場景：** 你解決了一個複雜的問題

**做法：**
1. 將解決方案記錄到專案文件中
2. 下次遇到類似問題時參考
3. 分享給其他團隊成員

**範例：**

在 `docs/solutions/` 下建立文件：

```markdown
# 解決 N+1 查詢問題

## 問題描述
在取得文章列表時，系統會為每篇文章查詢一次作者資訊，
導致性能問題。

## 解決方案
使用 JOIN 查詢一次性取得所有資料：

```sql
SELECT articles.*, users.name as author_name
FROM articles
LEFT JOIN users ON articles.author_id = users.id
```

**結果：** 查詢時間從 2000ms 降至 50ms

## 5.6 學習提問的藝術

**技巧一：先問「為什麼」**
```text
為什麼這段程式碼會造成記憶體洩漏？
為什麼應該使用 useCallback 而不是普通函式？
```

**技巧二：要求多個選項**
```text
實現使用者認證有哪些不同的方式？
各自的優缺點是什麼？
```

**技巧三：要求解釋**
```text
這段程式碼是如何運作的？
你能詳細解釋這個演算法嗎？
```

# 6. 常見問題

## Q1：我可以直接使用 AI 生成的程式碼嗎？

**A：** 不可以，不能直接使用。它需要審查和測試。

AI 生成的程式碼可能有以下問題：
- 邏輯錯誤或邊界情況處理不當
- 不符合專案的程式碼規範
- 安全風險
- 性能最佳化不足

你需要：
- 仔細閱讀生成的程式碼
- 理解其邏輯
- 測試不同場景
- 確認它遵循專案規範

## Q2：如果 AI 誤解了我的意思怎麼辦？

**A：** 及時糾正，重新描述需求。

```text
不是這個意思。我的意思是...
這個理解是不正確的。應該是...
讓我重新描述一下需求...
```

如果糾正幾次後仍然不正確，你可以：
- 提供更多上下文
- 給出具體的程式碼範例
- 將任務拆分為更小的部分

## Q3：遇到 AI 無法解決的問題怎麼辦？

**A：** AI 不是萬能的。有些問題仍然需要你自己解決。

AI 可能無法解決的問題：
- 非常新的技術（AI 的知識有截止日期）
- 你們團隊獨有的業務邏輯
- 需要存取外部系統的問題
- 複雜的性能最佳化問題

這時候，你需要：
- 閱讀官方文件
- 搜尋相關的解決方案
- 請教有經驗的同事
- 在社群中提問

## Q4：如何判斷 AI 的建議是否合理？

**A：** 用你自己的經驗和知識來判斷。

評估標準：
- 是否遵循最佳實踐
- 是否考慮了邊界情況
- 是否有潛在的安全風險
- 是否適合專案的技術堆疊
- 性能是否可接受

如果不確定，你可以：
- 請 AI 解釋為什麼建議這種做法
- 要求提供替代方案
- 諮詢團隊成員

## Q5：團隊應該如何使用 AI 協作？

**A：** 建立共享的規範和共享的知識庫。

團隊協作建議：
- 共用專案的 `CLAUDE.md` 設定
- 統一程式碼規範和風格
- 記錄常見問題的解決方案
- 定期分享有用的提示詞
- 在程式碼審查時檢查 AI 生成的程式碼

## Q6：如何避免過度依賴 AI？

**A：** 保持學習和思考。AI 是助手，不是替代品。

建議：
- 理解 AI 生成的程式碼，而不是盲目複製
- 主動學習你不了解的概念
- 定期複習基礎知識
- 先嘗試自己解決問題，再用 AI 驗證
- 參與程式碼審查，從他人的經驗中學習
